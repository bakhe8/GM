# سياسة العمل بعد `v1.0.0`

هذه الوثيقة تثبت طريقة تطوير النسخ القادمة بدون فقدان خط الأساس الذي تم اعتماده في `v1.0.0`.

## الفروع المعتمدة

- `main`
  خط التطوير الرئيسي للنسخ القادمة بعد الإصدار الأول.

- `release/v1.0.x`
  فرع صيانة مخصص فقط للإصلاحات الحرجة التي يجب أن تعود إلى النسخة `1.0.x`.

- `feature/<name>`
  أي تطوير جديد أو تحسين أو تجربة يجب أن يبدأ من فرع مستقل بهذا النمط.

## قواعد العمل

1. لا يتم العمل مباشرة على `main`.
2. لا يتم تعديل `v1.0.0` إلا عند وجود خلل حرج يمس التشغيل أو البيانات أو الأمان.
3. أي تحسين بصري أو تنظيمي أو توسعة وظائف يذهب إلى فرع `feature/<name>`.
4. عند اكتمال العمل على أي `feature` يتم دمجه إلى `main` فقط بعد نجاح:
   - `dotnet build -c Release`
   - `dotnet test -c Release`
   - smoke test مختصر
5. إذا ظهر خلل حرج في النسخة الأولى، يبدأ الإصلاح من `release/v1.0.x` ثم يدمج أيضًا إلى `main` حتى لا نفقد التوافق.

## ترقيم الإصدارات

- `1.0.1`, `1.0.2`, ...
  إصلاحات حرجة فقط على فرع `release/v1.0.x`

- `1.1.0-preview.1`, `1.1.0-preview.2`, ...
  نسخ تطويرية غير نهائية أثناء بناء أو مراجعة الواجهة الجديدة والوظائف التابعة لها

- `1.1.0-rc.1`, `1.1.0-rc.2`, ...
  نسخ مرشحة للإطلاق بعد اكتمال الواجهات الأساسية ومراجعتها وظيفيًا وبصريًا

- `1.1.0`
  تحسينات وميزات جديدة بدون كسر

- `2.0.0`
  تغييرات كبيرة أو تغييرات تكسر السلوك السابق

## خطوات البدء في أي تطوير جديد

1. تحديث `main` محليًا:
   `git checkout main`
   `git pull`

2. إنشاء فرع جديد:
   `git checkout -b feature/<name>`

3. تنفيذ التعديل واختباره.

4. رفع الفرع:
   `git push -u origin feature/<name>`

5. بعد الاعتماد، يدمج الفرع إلى `main`.

## خطوات الإصلاح الحرج على النسخة الأولى

1. الانتقال إلى فرع الصيانة:
   `git checkout release/v1.0.x`
   `git pull`

2. إنشاء فرع إصلاح:
   `git checkout -b fix/<name>`

3. بعد الاختبار، يدمج الإصلاح إلى:
   - `release/v1.0.x`
   - `main`

- baseline المعتمد: `v1.0.0`
- الوسم المرجعي: `v1.0.0`

## الحالة الحالية

- `main` يمثل حاليًا الحالة المدموجة حتى دفعة `v1.1-kickoff` عند commit `d855be1`
- الفرع النشط الآن: `feature/v1.1-operational-polish` وتُثبت حالته الحالية كـ `v1.1.0-preview.4` بعد دفعة اختصار الواجهات والتنظيف البنيوي
- فرع `release/v1.0.x` يبقى مخصصًا فقط لأي hotfix يعود إلى خط `1.0.x`
- النسخة التطويرية المعروضة داخل التطبيق على هذه الشجرة: `v1.1.0-preview.4`
- لا يُستخدم `v1.0.1` لمسار الواجهة الجديدة؛ يبقى محجوزًا فقط لأي hotfix حرج على خط `v1.0.x`

## بوابة الترقية بين الإصدارات

- `v1.1.0-preview.2` تمثل baseline نظيفة بعد إقفال جولة stabilization الأساسية
- `v1.1.0-preview.3` تمثل baseline تجميد بعد تغييرات workflow/UI الواسعة وإلغاء مسار النقض
- `v1.1.0-preview.4` تمثل baseline تجميد بعد جعل السجل الزمني محور العمل، اختصار واجهة الطلبات، وتنظيف الملفات الكبيرة
- ننتقل إلى preview لاحقة فقط إذا ظهرت دفعات تطويرية مؤثرة جديدة بعد هذه baseline
- ننتقل إلى `v1.1.0-rc.1` فقط بعد:
  1. اكتمال stabilization
  2. جولة UAT واقعية نظيفة
  3. عدم وجود مشاكل مانعة مفتوحة
