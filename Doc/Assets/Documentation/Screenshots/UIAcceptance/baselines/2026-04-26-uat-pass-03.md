# UAT Pass 03 — Native Windows and Missing External Files

Date: 2026-04-26

## Scope
- Continue UAT on external-file and print flows.
- Verify whether the UI tool can see and reason about native Windows print UI, not only WPF-owned dialogs.
- Verify how the product behaves when an expected response document is missing on disk.

## What We Exercised
1. Opened `الضمانات`
2. Opened `سجل الضمان`
3. Triggered `طباعة السجل`
4. Observed a native print surface titled `GuaranteeManager - Print`
5. Switched to `الطلبات`
6. Temporarily removed the `Workflow\\Responses` directory from its expected path
7. Triggered `فتح الرد` on the currently selected request
8. Observed the product response, app log behavior, and final session cleanliness

## What We Learned
- The print window is **not** hosted by the WPF process.
- It appears as a top-level native/system window:
  - Title: `GuaranteeManager - Print`
  - Class: `ApplicationFrameWindow`
- Earlier tooling focused too narrowly on the app process, so it could miss this class of windows.

## Tool Improvements Made In This Pass
- `Windows` now includes **related external windows**, not only windows owned by the app process.
- `Probe` now:
  - reports `ActiveDialogTitle = GuaranteeManager - Print` when this window is open
  - marks `HasExternalForegroundWindow = true`
  - captures a **desktop screenshot** instead of only the app window when a related external window is active
- Explicit title targeting now works on native windows such as:
  - `GuaranteeManager - Print`
- We verified direct element discovery inside the print surface, including:
  - `PrintButton`
  - `CloseButton`
  - close/title-bar button
- `Click`, `SetField`, and `Key` now use the same **smart scope-root resolution** that `Elements` already used.
  - This closed a real tool gap where `Elements -WindowAutomationId Requests.Workspace` worked but `Click -WindowAutomationId Requests.Workspace ...` could still time out as if the workspace were a top-level window.
- The tool now has a reliable path for documenting product dialogs opened from **internal workspace roots**, not only from the main shell window or native top-level dialogs.

## Product Fix Confirmed In This Pass
- Earlier in UAT, missing response files could surface only through the generic top-level system alert and `App_DispatcherUnhandledException`.
- After the product fix in the request/guarantee coordinators, the same missing-file scenario now produces a **local friendly dialog**:
  - Title: `رد البنك`
  - Message: `تعذر فتح رد البنك. الملف غير موجود.`
- We also confirmed that the app log does **not** append a fresh unhandled exception for this repeated scenario after the fix.

## Behavioral Result
- The tool successfully detected and inspected the native print window.
- `DialogAction -WindowTitle "GuaranteeManager - Print" -Text "Cancel"` now closes the native print window successfully after the follow-up hardening in this pass.
- A direct click on the print dialog `Cancel` button also closes it successfully, which gives us a reliable fallback path.
- The product now handles missing response documents gracefully from `الطلبات` without falling back to a generic system alert.
- Final probe after close:
  - `ActiveDialogTitle = null`
  - `HasExternalForegroundWindow = false`
  - `OpenWindowCount = 1`
