# UAT Pass 04 — Confirmation Dialog Readiness and Closure Reliability

Date: 2026-04-26

## Scope
- Continue controlled UAT from the clean `v1.1.0-preview.2` baseline.
- Re-test the `NewGuaranteeDiscard` path after a real failure in the confirmation step.
- Verify whether the problem was in the product or in the UI tool's interaction timing.

## What Happened
During the first `preview.2` UAT run, the scenario failed at:

1. Open `إجراء جديد`
2. Type a guarantee number
3. Press `إلغاء`
4. Show `تأكيد الإغلاق`
5. Try to confirm close

The tool then timed out waiting for `تأكيد الإغلاق` to close.

## What We Verified
- A live `Probe` and `Windows` read showed that this was **not** a false timeout:
  - `تأكيد الإغلاق` was genuinely still open
  - `إجراء جديد` was still open behind it
- This means the original issue was not just observation noise; the confirmation click did not always land effectively.
- The user's diagnosis was correct: the confirmation action could happen **too quickly**, before the dialog felt fully ready to respond.

## Tool Improvements Made In This Pass
- Added dialog-action hardening directly in the shared UI automation module:
  - `Wait-UiElementReady`
  - `Wait-UiWindowGone`
  - `Invoke-UiDialogActionButton`
- `DialogAction` no longer treats "button was found and clicked" as success.
  - It now waits for:
    - the dialog to be foregrounded
    - the target button to be visible and enabled
    - an actual close signal after the action
- The new dialog action path retries with progressively safer strategies:
  1. invoke pattern / default action
  2. focused `Space`
  3. focused `Enter`
  4. direct mouse click fallback
- `run_ui_acceptance.ps1` now uses the same hardened dialog action path instead of a fast one-shot confirmation click.

## What We Exercised After The Fix
1. Used `DialogAction` against the already stuck `تأكيد الإغلاق` dialog
2. Verified that the dialog closed successfully
3. Verified in a **separate sequential probe** that the session returned to:
   - `ActiveDialogTitle = null`
   - `OpenWindowCount = 1`
4. Re-ran:
   - `NewGuaranteeDiscard`
5. Re-ran:
   - `All`

## Behavioral Result
- The confirmation dialog can now be resolved reliably after appearing.
- The `NewGuaranteeDiscard` scenario passed successfully.
- The full `All` acceptance run also passed successfully after the hardening.
- The remaining difference is performance, not correctness:
  - `DialogAction` became slightly slower
  - but more trustworthy, which is the right trade here for controlled UAT

## Notes
- This pass is a good example of the right workflow:
  - the product exposed a behavior
  - the user interpreted it correctly
  - the tool was then improved so we do not repeat the same blind spot
- The issue was **not** solved by reducing visibility or skipping the dialog.
  - It was solved by making the tool behave more patiently and verify actual closure.
