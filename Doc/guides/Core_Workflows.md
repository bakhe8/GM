# Core Workflows

**تاريخ الإنشاء:** 2026-04-28  
**الغرض:** تعريف العمل الحقيقي للبرنامج على مستوى المهام، لا على مستوى أسماء الشاشات.

## كيف نستخدم هذا الملف

هذا الملف ليس دليل استخدام.  
هو مرجع تصميم تشغيلي يجيب عن سؤال:

**ما المهام الأساسية التي يجب أن تبنى الواجهة حولها؟**

## المهام الأساسية المعتمدة

### 1. بدء اليوم وتحديد الأولوية

- **الهدف:** معرفة ما الذي يحتاج انتباهًا الآن
- **نقطة الدخول الرئيسية:** `Dashboard`
- **النتيجة الصحيحة:** الانتقال إلى ضمان أو queue أو surface محددة بسياق واضح

### 2. مراجعة الطلبات اليومية المعلقة

- **الهدف:** التقاط العمل الجاري الآن على البنك
- **نقطة الدخول الرئيسية:** `Requests`
- **النتيجة الصحيحة:** تنفيذ الفعل التالي على الطلب ثم البقاء داخل queue أو الانتقال للضمان عند الحاجة

### 3. فتح ضمان واتخاذ قرار

- **الهدف:** فهم حالة ضمان واحد ثم إطلاق الفعل المناسب
- **نقطة الدخول الرئيسية:** `Guarantee File`
- **النتيجة الصحيحة:** قرار أو طلب أو مراجعة أو انتقال إلى history/inquiry

### 4. تسجيل رد بنك

- **الهدف:** إغلاق أو تحديث طلب بناءً على الرد
- **نقطة الدخول الرئيسية:** `Requests`
- **نقطة الدخول الثانوية:** `Guarantee File`
- **النتيجة الصحيحة:** تحديث الطلب، تحديث السياق، ظهور الخطوة التالية الصحيحة

### 5. فتح خطاب الطلب أو رد البنك

- **الهدف:** الوصول إلى المستند الخارجي المطلوب بسرعة
- **نقطة الدخول الرئيسية:** `Requests`
- **نقطة الدخول الثانوية:** `HistoryDialog` / `OperationalInquiryDialog`
- **النتيجة الصحيحة:** فتح المستند مع feedback واضحة، دون فقدان السياق

### 6. إنشاء طلب جديد على ضمان مؤهل

- **الهدف:** إطلاق workflow جديدة على الضمان المناسب
- **نقطة الدخول الرئيسية:** `Guarantee File`
- **نقطة الدخول الثانوية:** `Requests`
- **النتيجة الصحيحة:** إنشاء الطلب، تحديث العداد، وتغيير الأفعال الظاهرة حسب الحالة الجديدة

### 7. فهم التاريخ أو الجواب التشغيلي

- **الهدف:** الانتقال من الفهم إلى الفعل
- **نقطة الدخول الرئيسية:** `HistoryDialog` / `OperationalInquiryDialog`
- **النتيجة الصحيحة:** ظهور الدليل والقرار التالي في نفس السياق

### 8. تصدير أو طباعة أو حفظ ناتج

- **الهدف:** إنتاج مخرج تشغيلي واضح
- **نقطة الدخول الرئيسية:** `Reports` أو الحوارات الثقيلة
- **النتيجة الصحيحة:** إنشاء الناتج أو طباعته مع feedback صحيحة وبدون ضياع

### 9. حماية التشغيل المحلي

- **الهدف:** backup / restore / reseed / تشغيل محلي
- **نقطة الدخول الرئيسية:** `Settings`
- **النتيجة الصحيحة:** تنفيذ الإجراء ثم refresh صحيح للـ shell والبيانات

## مبدأ Canonical Flow

لكل مهمة من المهام السابقة يجب أن يكون هناك:

1. **نقطة دخول رئيسية**
2. **خطوة تالية واضحة**
3. **سياق محفوظ**
4. **إغلاق طبيعي**

إذا ظهرت نفس المهمة من أكثر من surface:

- نسمح بذلك
- لكن يجب أن تبقى **النسخة الرئيسية للمسار** واضحة
- ويجب ألا تتحول الواجهات إلى نسخ متنافسة من نفس الوظيفة

## السطوح الرئيسية بحسب الدور

| السطح | دوره الحقيقي |
|---|---|
| `Dashboard` | ابدأ يومك وحدد الأولوية |
| `Requests` | queue تنفيذ يومية |
| `Guarantee File` | فهم + قرار + إطلاق تنفيذ |
| `HistoryDialog` | فهم زمني يقود إلى فعل |
| `OperationalInquiryDialog` | جواب تشغيلي يقود إلى فعل |
| `Reports` | إخراج وتشغيل batch |
| `Settings` | تشغيل محلي وصيانة |

## الخلاصة

الواجهات لا يجب أن تُبنى حول:

- “ما الذي يمكن عرضه هنا؟”

بل حول:

- “ما المهمة التي يجب أن تُنجز هنا بأوضح شكل؟”
