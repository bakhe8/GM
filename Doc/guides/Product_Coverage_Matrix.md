# Product Coverage Matrix

**تاريخ الإنشاء:** 2026-04-28  
**الغرض:** منع إعادة الجولات المشبعة، وحصر إعادة الفحص في حالات `delta / bug / risky workflow` فقط.

## لماذا هذا الملف

بعد جولات `Settings` و`Requests` و`Guarantee Operational` و`Guarantee Authoring` و`cross-entry`، صارت التغطية العامة للمنتج عالية بما يكفي ليتحول العمل من:

- جولات واسعة متكررة

إلى:

- فحص موجّه بالدلتا
- أو bug report واضح
- أو مسار عالي الخطورة تغير فيه الكود

هذا الملف هو المرجع التنفيذي الذي يمنعنا من:

- إعادة pass clean على surface مستقرة
- استخدام الأداة في كل فحص حتى عندما لا تضيف evidence جديدة
- خلط `التحقق المفيد` مع `الحركة الزائدة`

## معاني الحالة

- `مشبع`
  - المنطق اليومي الأساسي مغطى، وإعادة الجولة لا تتم إلا مع delta واضحة
- `مستقر`
  - السطح جيد، لكن ما زال يستحق فحصًا موجّهًا إذا تغيرت نقطة قريبة منه
- `مركز`
  - لا نعيد الجولة كاملة، بل نتحقق من جزء بعينه فقط

## سياسة استخدام الأداة من الآن

### نستخدم الأداة فقط إذا كان هناك:

1. نافذة نظام أو نافذة خارجية
2. غموض في الحالة أو تناقض بين الشاشة و`ShellState`
3. ملف خارجي أو رجوع بسياق أو dialog native
4. تحقق end-to-end بعد إصلاح workflow
5. حاجة إلى evidence بصرية/زمنية لا يكفي فيها فحص الكود

### لا نستخدم الأداة افتراضيًا إذا كان:

1. الفحص منطقيًا من الكود واضحًا
2. السطح مستقرًا ولم يتغير
3. المطلوب مجرد تأكيد delta صغيرة يمكن عزلها بفحص مركز
4. pass سابقة clean غطت نفس المسار ولم يدخل عليه تغيير

## مصفوفة التغطية الحالية

| السطح | الحالة | ما تم إقفاله | متى نعيد الفحص | مستوى الأداة |
|---|---|---|---|---|
| `Settings` | مشبع | feedback للنسخ/الفتح، refresh بعد restore/reseed، تطبيع مسار backup، دورة backup/restore كاملة | إذا تغيرت `restore/reseed/backup` أو dialogs النظامية أو session refresh | `انتقائي` |
| `Requests` | مشبع | `رد البنك`، `فتح الرد`، `إلحاق الرد`، `فتح الضمان`، metadata للخطوة التالية، صدق `ShellState` | إذا تغير response workflow أو context handoff أو file opening | `انتقائي` |
| `Guarantee Operational Path` | مشبع | `HistoryDialog` بسياق الطلب، `OperationalInquiry -> export report`، `History -> print cancel` | إذا تغيرت dialogs الثقيلة أو save/print integration أو next-step logic | `انتقائي` |
| `Guarantee Authoring Path` | مشبع | جديد/تعديل، discard prompts، restore focus بعد الحفظ، تمديد/إفراج/تخفيض/استبدال/نقض/تسييل/تحقق | إذا تغيرت قواعد الأهلية أو authoring dialogs أو post-save refresh | `انتقائي` |
| `Guarantee File structure` | مركز / WIP | بداية الملف وخريطة الملف أُغلقت، ووسط الملف صار يقسم `الطلبات/الخط الزمني/المخرجات/المرفقات` حسب سؤال العمل | بعد إغلاق WIP الحالي أو إذا تغيرت bindings / focus / section navigation | `build/test + فحص مركز عند الحاجة` |
| `Dashboard / Notifications cross-entry` | مشبع كـ cross-entry فقط | فتح الضمان بسياق صحيح، carry context إلى `Requests/Notifications`، return-to-work clean | إذا تغيرت navigation أو handoff أو session persistence | `انتقائي` |
| `Reports` | مستقر | feedback لإنشاء التقرير وفتحه، metadata أكثر صدقًا، حفظ فعلي clean | إذا تغير export workflow أو dialogs الحفظ أو latest-output logic | `انتقائي` |
| `Banks` | مستقر | feedback هادئة لأفعال النسخ | إذا تغير clipboard/status behavior أو daily actions | `عند الحاجة فقط` |
| `Guarantees list/search/filter` | مستقر | الحفاظ على الاختيار عند بقاء السجل مرئيًا بعد البحث/الفلتر | إذا تغيرت `Refresh()` أو sorting/filter/search logic | `فحص مركز` |
| `Backup / restore full cycle` | مشبع لكن ثقيل | إنشاء واسترجاع نسخة من نفس الملف حتى `data-reset` | فقط بعد تغييرات `Settings` أو `Shell refresh` أو dialogs النظامية | `فقط عند الدلتا` |

## ما الذي نتوقف عن فعله من الآن

1. لا نعيد جولة عامة على surface `مشبعة` بلا تغيير كود أو bug report
2. لا نكرر pass clean فقط لإنتاج توثيق جديد
3. لا نشغل الأداة تلقائيًا في كل فحص
4. لا نعيد UAT واسعة على مسار stable إذا كان المطلوب مجرد delta صغيرة

## ما الذي نفعله بدلًا من ذلك

1. نبدأ من `التغيير` لا من `السطح`
2. نحدد:
   - ما الملف الذي تغيّر؟
   - ما المسار الذي تأثر؟
   - هل يحتاج فحصًا منطقيًا أم end-to-end؟
3. نقرر مستوى التحقق:
   - `فحص كود فقط`
   - `فحص مركز`
   - `فحص بالأداة`
   - `فحص end-to-end كامل`

## قاعدة العمل الحالية

أي surface في هذه المصفوفة لا تُعاد جولتها إلا إذا وقع واحد من التالي:

1. **Delta**
   - تغير كود يمس هذا السطح أو تدفقه
2. **Bug**
   - ظهر سلوك جديد أو report يمكن نسبته لهذا السطح
3. **Risk**
   - التدفق يتضمن:
     - dialogs نظامية
     - windows خارجية
     - ملفات فعلية
     - أو handoff سياقي حساس

إذا لم يوجد `Delta / Bug / Risk`:

- لا نعيد الجولة
- وننتقل مباشرة إلى العمل التالي في البرنامج

## الحكم الحالي

الوضع الآن ليس:

- `نحتاج تغطية أوسع`

بل:

- `نحتاج انضباطًا أعلى في اختيار ما نعيد فحصه`

وهذا يعني أن الأداة من الآن تدخل **وضع خدمة التحقق الموجّه**، لا **وضع الوجود الدائم في كل خطوة**.
